Есть ряд непоняток с TimeRCD

1) Способ маскирования и `patch_size`.
[Здесь](https://github.com/thu-sail-lab/Time-RCD/blob/tsb-ad-integration/model_wrapper.py#L426) ставят `config.patch_size=16`, однако `TimeRCD.py` обращается к `collate_fn` из `TimeRCD_pretrain_multy.py`, где такая строчка:
```
patch_size = default_config.ts_config.patch_size
```
А `default_config.ts_config.patch_size = 4`, так как `default_config` -- это конфиг по умолчанию, инпортируемый как статичная переменная из конфигурации. Т.е. как будто бы вообще не важно, через какой `patch_size` происходит маскирование, и он может отличаться от того, что указан в конфиге детектора.

Кроме того, функция `create_random_mask имеет две реализации`. В исходном коде есть две версии одной строчки:
* В `training.py` -- `masked_time_series[mask_expanded] = 0.0`
* В `TimeRCD_pretrain_multi.py` -- `masked_time_series[mask_expanded] = torch.randn_like(masked_time_series[mask_expanded]) * 0.1`

Последнее -- типа для визуализации в файле `TimeRCD.py` в методе `TimeRCDPretrainTester.test_model`, однако там `masked_time_series` ВООБЩЕ нигде не используется -- предикт строится на основе `time_series`, т.е. смысла как будто бы никакого не несет.

Спрашивал перплексити:
```
Проблемы supervised TimeRCD python

В create_random_mask() — маски только на валидных!
masked_time_series[mask_expanded] = 0.0  # Не randn! 
Слишком простая маскировка — модель "учит" заполнять нулями, а не контекстом

Аномалии не влияют на recon — модель игнорирует их в favor нормальных паттернов
```

Проверка влияния маски на `anomaly_loss` не показала значимых результатов

2) Датасеты и модели.
Есть версия reconstruction only, есть даже датасет под нее, но примеров использования в коде работы нет

3) Zero-shot под внешние датасеты.
Как сказал Михаил, не пригодится.

4) anomaly_scores vs anomaly_logits
См. файл `TimeRCD.py`, а именно на функцию `zero_shot`. Она возвращает две величины: `anomaly_probs` на выходе головы с софтмаксом для положительного класса (классические вероятности) и `anomaly_logits` -- разница между выходами модели для положительного и отрицательного класса, т.е. `anomaly_logits[..., 1] - anomaly_logits[..., 0]`. Можно трактовать как уверенность модели в аномалии (а-ля decision_function). Предикты из логитов получаются [так](https://github.com/thu-sail-lab/Time-RCD/blob/tsb-ad-integration/main.py#L305): порог равен `mean + 3 * std` по этим логитам.

5) Аномалии по реконструкции. Как раз о чем говорил Михаил: `anomaly_probs = ((reconstructed - time_series) ** 2)[mask_expand]``. В такой формулировке действительно не зачем обучать бошку.

На данный момент код выстроен так, что можно указать обучение `anomaly_head`: если `False`, то anomaly_loss просто всегда будет нулем. Кроме того, для бенчмаркинга можно выставить, как будут считаться `anomaly_scores`: как `"probs"` или как `"logits"`. В случае, когда не обучаем `anomaly_head`, доступно только `"logits"`.

Что заметил при обучении:
* Увеличения `patch_size` очень сильно улучшает реконструкцию, размерность модели не влияет не так сильно (разница между 256 и 512 `d_model` недраматическая)
* Увеличения `num_heads` требует больше эпох для обучения
* Маска -- будь то нулевая, будь то случайная -- роли не играет
* В сеттинге с фиттингом `anomaly_head` не играет роли ни маскирование, ни жир модели, ни вес `anomaly_loss` -- вероятности пляшут в одном диапазоне около 0.2 при любом раскладе, а лосс падает в плато на первых 20-ти эпохах. Возможно, сработает, если сделать только крайне жирную модель.

В коде НЕТ реализации token-level разметки, где токен -- это патч, но во всех кодах и файлах датасета разметка point-level -- для каждого временного шага
Есть два ключевых утверждения:
1) We treat each time window of a time series as an input token
2) It provides token-level annotations for a diverse spectrum of anomalies / We generate token-level binary labels per channel
Итог по token-level разметке: Это техника генерации синтетических данных (аномалии всегда выровнены по границам патчей), а не формат хранения и не способ обучения модели. Поэтому в данных вы видите обычные поточечные лейблы. Путаница возникла из-за того, что авторы в статье называют эту технику генерации "token-level annotations", что технически верно на этапе генерации (не факт), но в итоговом датасете это обычные поточечные лейблы.
